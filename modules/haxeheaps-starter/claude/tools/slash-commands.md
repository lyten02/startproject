# Slash Commands в Claude Code

## Инструкция для Claude

Когда пользователь просит создать или изменить slash команду, выполни следующие шаги:

### 1. Проверка существующих команд

**Перед созданием ОБЯЗАТЕЛЬНО проверь**, существует ли уже такая команда:

```bash
# Проверить проектные команды
ls -la .claude/commands/

# Проверить пользовательские команды
ls -la ~/.claude/commands/

# Найти конкретную команду по имени
find .claude/commands ~/.claude/commands -name "*.md" 2>/dev/null | grep -i "<имя_команды>"
```

**Правила определения действия:**

- Если команда **существует** → предложи **изменить** её
- Если команда **не существует** → предложи **создать** новую
- Если существуют похожие команды → сообщи пользователю и уточни, что именно нужно

### 2. Структура директорий

| Тип            | Путь                         | Описание                     |
| -------------- | ---------------------------- | ---------------------------- |
| Проектные      | `.claude/commands/`          | Общие для команды, в git     |
| Личные         | `~/.claude/commands/`        | Только для пользователя      |
| С неймспейсом  | `.claude/commands/frontend/` | Группировка по категориям    |

**Приоритет**: Проектные команды имеют приоритет над личными с тем же именем.

### 3. Формат файла команды

Файл: `.claude/commands/<имя-команды>.md`

```markdown
---
description: Краткое описание (показывается в /help)
argument-hint: [arg1] [arg2]
allowed-tools: Bash(git:*), Read, Write
model: claude-3-5-haiku-20241022
---

Текст промпта для команды.

Используй $ARGUMENTS для всех аргументов или $1, $2 для конкретных.

## Контекст (опционально)

- Статус: !`git status`
- Файл: @src/example.js
```

### 4. Переменные

| Переменная     | Описание                     | Пример                                    |
| -------------- | ---------------------------- | ----------------------------------------- |
| `$ARGUMENTS`   | Все аргументы как строка     | `/cmd foo bar` → `"foo bar"`              |
| `$1`, `$2`...  | Позиционные аргументы        | `/cmd foo bar` → `$1="foo"`, `$2="bar"`   |
| `` !`cmd` ``   | Вывод bash команды           | `` !`git status` ``                       |
| `@path`        | Содержимое файла             | `@src/main.js`                            |

### 5. Поля frontmatter

| Поле                       | Назначение                                 | Обязательность |
| -------------------------- | ------------------------------------------ | -------------- |
| `description`              | Описание для /help                         | Рекомендуется  |
| `argument-hint`            | Подсказка аргументов                       | Опционально    |
| `allowed-tools`            | Разрешённые инструменты                    | Опционально    |
| `model`                    | Конкретная модель                          | Опционально    |
| `disable-model-invocation` | Запретить вызов через SlashCommand tool    | Опционально    |

### 6. Примеры создания

**Простая команда:**

```markdown
---
description: Запустить тесты для модуля
argument-hint: [module-name]
---

Запусти тесты для модуля $1 и покажи результаты.
```

**Команда с контекстом:**

```markdown
---
description: Создать коммит по conventional commits
allowed-tools: Bash(git:*)
---

## Контекст

- Изменения: !`git diff --staged`
- Статус: !`git status`

Создай коммит в формате conventional commits (feat:, fix:, docs: и т.д.)
```

**Команда с файловым контекстом:**

```markdown
---
description: Ревью компонента
argument-hint: [file-path]
---

Проведи код-ревью файла @$1:

- Проверь best practices
- Найди потенциальные баги
- Предложи улучшения
```

### 7. Алгоритм действий

```text
1. Пользователь запрашивает создание/изменение команды
   ↓
2. Проверить существующие команды (ls, find)
   ↓
3. Если существует:
   - Прочитать текущее содержимое
   - Показать пользователю
   - Предложить изменения
   ↓
4. Если не существует:
   - Убедиться, что директория .claude/commands/ существует
   - Создать файл с правильным форматом
   ↓
5. Подтвердить создание/изменение пользователю
```

### 8. Проверка после создания

После создания команды проверь:

- Файл создан в правильной директории
- Синтаксис frontmatter корректен (YAML)
- Команда доступна через `/help` или автокомплит
