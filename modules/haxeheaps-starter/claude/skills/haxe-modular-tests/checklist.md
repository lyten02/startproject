# Test review checklist

Прогнать перед тем, как сообщить «тест готов». Каждый пункт — да/нет.

## Файл и класс

- [ ] Имя файла совпадает с именем класса.
- [ ] Класс `extends utest.Test` (или `extends Test` при `import utest.Test`).
- [ ] Имя оканчивается на `Spec` (предпочтительно) или начинается с `Test` (legacy). **Не** оканчивается на `Fixture` — такие макрос исключит.
- [ ] Файл лежит в правильной папке:
  - проектный класс → `test/<pkg>/…`
  - класс модуля → `modules/<mod>/test/unit/…`

## Импорты

- [ ] Нет `import h2d.*` / `import hxd.*` / `import h3d.*` (правило D из `CLAUDE.md`).
- [ ] Нет `hxd.Key.*` (правило H). Моки ввода — через `GameAction`.
- [ ] Ресурсы читаются через typedef, не через `sys.io.File.getContent`.

## Структура

- [ ] `setup()` только инициализирует, без `Assert.*`.
- [ ] Каждый `test*` метод независим (не полагается на порядок или состояние от других).
- [ ] Поля класса инициализируются в `setup()`, не в объявлении (если это не константы).
- [ ] Тесты публичного контракта, не приватных полей.

## Ассерты

- [ ] Использованы нужные `Assert.*`: `equals/same/isTrue/isFalse/raises/floatEquals/match`.
- [ ] Для плавающей точки — `floatEquals` с осмысленным epsilon, не `equals`.
- [ ] Для структурной эквивалентности — `same`, не поэлементные `equals`.
- [ ] Для ожидаемого исключения — `raises`, не `try/catch`.

## Прогон

- [ ] `python build.py test --ai` → `"ok": true` (или `"ok": false` с осмысленным `failing_lines` при red-стадии TDD).
- [ ] Если тест для модуля — также `python build.py test --module=<mod> --ai` возвращает `ok: true`.
- [ ] Общее число ассёртов в отчёте ≥ прежнего (новый тест не потёр старые).

## Регистрация

- [ ] `TestMain.hx` **не** менялся — макрос подхватит автоматически.
- [ ] Ни в какой список/массив класс вручную не добавлен.
